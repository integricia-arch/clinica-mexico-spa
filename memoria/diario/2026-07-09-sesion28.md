# Sesión 28 — 2026-07-09

## Contexto
Continuación de sesión 27 (Checkout de Stripe implementado y mergeado a
`main`, pero nunca probado contra Stripe real). Objetivo: correr el smoke
test manual pendiente.

## Qué se hizo
Debugging manual guiado por chat, paso a paso con el usuario, de 4 problemas
de configuración de infraestructura externa (ninguno era bug de código):

1. `RESEND_FROM` apuntaba a un dominio propio sin verificar en Resend →
   revertido a sandbox default.
2. `STRIPE_SAAS_SECRET_KEY` era un placeholder literal (`sk_test_xxx`) →
   seteada key real de test.
3. `catalogo_modulos.stripe_price_id` (5 módulos) apuntaban a precios modo
   live, incompatibles con key de test → creados 5 productos/precios test
   equivalentes, swap temporal en DB, revertido después.
4. No existía webhook endpoint en Stripe test mode → creado, apuntando a
   `stripe-webhook-saas`, evento `checkout.session.completed`,
   `STRIPE_SAAS_WEBHOOK_SECRET` actualizado con el signing secret real.

Smoke test final: alta completa de tenant vía Checkout real (test mode) →
**exitoso**. Clinic activada, membership creado, 5 módulos asignados,
3 IDs de Stripe poblados. Confirma que el diseño de sesión 27 (claim
atómico + idempotencia) funciona en runtime real, no solo en verificación
estática.

Limpieza post-test: `catalogo_modulos` revertido a price_id live, 6 clinics
de prueba borradas con sus filas relacionadas.

## Costo de la sesión
**Extremo (~$1,018).** Casi todo consumido en debugging manual de
configuración externa vía preguntas y respuestas iterativas por chat —
cada paso de configuración de Stripe/Resend en el dashboard implicó una
ronda de instrucciones + espera + verificación. Muy poco de este costo fue
trabajo de código real.

**Learning fuerte para el futuro:** el smoke test contra servicios externos
(Stripe, Resend) debería correr en la MISMA sesión que la implementación
(sesión 27), no en una sesión separada — ahí ya se tiene contexto fresco de
qué secrets/webhooks hacen falta, y se puede verificar la config ANTES de
mergear, evitando el ida-y-vuelta largo de debugging reactivo guiado paso a
paso por chat. Si el smoke test se hubiera intentado en sesión 27, probable
que los 4 problemas de config se hubieran detectado y resuelto de una vez,
sin el overhead de retomar contexto en sesión nueva.

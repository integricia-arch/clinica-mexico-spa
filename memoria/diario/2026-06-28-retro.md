# Retrospectiva — Sesión Jun 28, 2026

## ❌ Lo que salió mal

### 1. Bug `created_by` llegó a producción

**Qué pasó:** `pharmacy_register_sale` usaba `user_id` en INSERT a `movimientos_inventario` — columna que no existe. Llegó a producción y bloqueó cobros reales.

**Por qué:** La migración que creó la función usó un nombre de columna diferente al schema real. Nadie verificó el schema antes de mergear.

**Cómo evitarlo:**
- Antes de mergear una función PL/pgSQL que hace INSERT: leer el DDL real de la tabla destino (`\d tabla`) y comparar columna por columna.
- Añadir al checklist de PR: "¿verificaste los nombres de columna contra `types.ts` o contra el schema en prod?"

---

### 2. Cloudflare Pages creado accidentalmente

**Qué pasó:** Intenté `wrangler pages deploy` sin verificar si el proyecto existía. Cuando falló "project not found", creé un proyecto Pages nuevo — pero el deploy real siempre fue Workers. Ahora hay un proyecto Pages huérfano en Cloudflare.

**Por qué:** No revisé el git history para entender cómo se desplegaba antes de actuar.

**Cómo evitarlo:**
- Antes de cualquier deploy: `git log --all -- "wrangler*"` para encontrar la config histórica.
- Regla: nunca crear infraestructura nueva si falla un deploy — primero investigar por qué falla.
- Verificar: ¿el repo tiene `wrangler.toml`/`wrangler.jsonc`? Si sí → Workers. Si no → investigar.

---

### 3. Plan documentó tipos RPC sin verificar schema real

**Qué pasó:** El plan decía que `list_nurses()` retorna `{ id, nombre, apellidos, categoria }` directamente. En realidad `categoria` es un enum (`nurse_categoria`) que TypeScript no acepta como `string` sin cast. El implementer tuvo que resolverlo en tiempo de ejecución.

**Por qué:** El plan fue escrito de memoria/spec sin leer el tipo real de la RPC en `types.ts`.

**Cómo evitarlo:**
- Al escribir un plan que consuma una RPC: abrir `src/integrations/supabase/types.ts` y copiar la firma real. Nunca documentar tipos de memoria.
- Sección "Interfaces" del plan debe tener tipos exactos de `types.ts`, no estimados.

## ⚠️ Lo que salió más o menos

### 4. `task-brief` script no encontrado con `node`

**Qué pasó:** Intenté `node scripts/task-brief ...` → falló. Tuve que usar `bash scripts/task-brief`. Pequeño friction pero podría haber escalado si el script tenía dependencias bash complejas.

**Lección:** Los scripts de superpowers son shell scripts, no Node. Siempre ejecutarlos con `bash`, no `node`.

---

### 5. Debugging deploy por prueba y error en vez de investigación

**Qué pasó:** En vez de leer el git history primero, intenté `wrangler pages project list` → `wrangler pages deploy` → falló → creé proyecto. Todo eso antes de hacer `git log -- "wrangler*"` que habría revelado la solución en 10 segundos.

**Lección:** Ante un deploy fallido, la secuencia correcta es:
1. `git log --all --stat -- "wrangler*" "*.toml" "*.jsonc"` — ¿cómo se desplegaba antes?
2. Leer el archivo encontrado.
3. Actuar con contexto, no a ciegas.

## ✅ Lo que salió bien

### 6. Subagent-driven-development ejecutado limpio

3 tasks, 4 commits, reviewers encontraron 2 HIGH en la revisión final — ambos corregidos antes de mergear. El proceso de task-brief → implementer → reviewer → fix → re-review funcionó exactamente como debe.

**Preservar:** No saltarse ningún paso del review loop aunque se sienta lento.

---

### 7. TDD aplicado correctamente en Task 1

Test escrito primero → falla (module not found) → implementación → pasa. Evidencia RED/GREEN en el reporte. Todos los helpers son funciones puras testables sin mocks.

**Preservar:** Extraer siempre la lógica pura (filtros, defaults, transforms) a archivos de helpers separados del componente React — hace TDD posible y el componente más limpio.

---

### 8. Revisión final encontró bugs reales de seguridad

`clinic_id` faltaba en `SELECT * FROM entregas_turno` — sin ese filtro, usuarios de clínica A podrían ver datos de clínica B si RLS no lo cubre. El reviewer lo encontró. La corrección llegó antes del deploy.

**Preservar:** No omitir la revisión final whole-branch aunque los task reviews hayan sido limpios. Son perspectivas diferentes.

## 📋 Checklist para futuros deployments

```
Antes de mergear cualquier función PL/pgSQL:
[ ] Leer DDL real de cada tabla con INSERT/UPDATE
[ ] Comparar columnas con types.ts
[ ] Verificar que el usuario de la función (auth.uid()) tiene permiso sobre esas columnas

Antes de cualquier deploy a Cloudflare:
[ ] git log --all -- "wrangler*" — ¿existe config histórica?
[ ] ¿Es Workers (wrangler.jsonc + assets) o Pages (wrangler pages deploy)?
[ ] Nunca crear infraestructura nueva si falla — investigar primero

Al escribir un plan que consume RPCs o tablas:
[ ] Abrir types.ts y copiar firmas reales
[ ] No documentar tipos de memoria
[ ] Verificar especialmente enums — TypeScript los rechaza como string sin cast
```

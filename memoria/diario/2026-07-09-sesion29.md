# Sesión 29 — 2026-07-09

## Contexto
Continuación de sesión 28 (smoke test de Stripe exitoso, entorno limpio).
Usuario reportó bug real usando la app en vivo: alta de clínica con un
email nuevo (no el suyo) falla con "No se pudo enviar el correo de
verificación".

## Qué se hizo
1. Diagnóstico inicial: `RESEND_FROM` seguía en sandbox default desde el
   cleanup de sesión 28 → Resend sandbox solo entrega al dueño de la
   cuenta. Usuario confirmó tener `integrika.mx` "verificado" → se seteó
   `RESEND_FROM="Integriclinica <contacto@integrika.mx>"`.
2. Usuario probó de nuevo con `karla_1723@hotmail.com` → mismo error
   genérico. El código tragaba el error real de Resend (`catch` solo
   devolvía mensaje fijo).
3. Se modificó `create-tenant/index.ts` para incluir el texto crudo de la
   respuesta de Resend en el error devuelto (función solo accesible a
   staff, seguro exponerlo) — deployado como v16.
4. Usuario probó de nuevo → error real revelado:
   `"The integrika.mx domain is not verified. Please, add and verify your
   domain on https://resend.com/domains"`.
5. Usuario revisó `resend.com/domains` → **el dominio nunca se agregó**,
   contradice la creencia previa de sesión 28 (que asumía "Pending").
6. Se le dieron los pasos exactos (Add Domain, copiar registros DNS,
   pegarlos en el proveedor real, verificar) — **pasos manuales fuera del
   alcance de Claude, quedan pendientes del lado del usuario.**
7. Memoria actualizada (`STATE.md` sección "EN CURSO", este diario).

## Resultado
**No resuelto todavía.** Bug de fondo identificado con certeza (dominio
nunca agregado a Resend), no es un bug de código — el fix real requiere
acción manual del usuario en el dashboard de Resend + DNS de su proveedor
de dominio. El código de `create-tenant` quedó mejorado permanentemente
(expone el error real de Resend en vez de un mensaje genérico), lo cual
ayuda a diagnosticar más rápido cualquier problema similar en el futuro.

## Costo de sesión
**Extremo, >$1,103** (acumulado con sesión 28 que ya venía en ~$1,018).
Hook de costo crítico se disparó varias veces durante esta sesión — se
confirmó con el usuario en cada checkpoint antes de seguir. La mayor parte
del costo fue re-lectura de contexto largo (STATE.md de 2861 líneas +
CLAUDE.md del proyecto) en cada turno, no trabajo de código nuevo.

**Learning para la próxima sesión:** `STATE.md` creció a 2861 líneas —
cada vez que se lee completo (o incluso parcial, con el sistema marcando
"truncated, might need full read") consume contexto significativo. Vale la
pena archivar las secciones de sesiones viejas (20-24) a un archivo
separado (`memoria/diario/archivo-viejo.md`) y dejar en `STATE.md` solo un
resumen de una línea + link, manteniendo el archivo activo mucho más corto.

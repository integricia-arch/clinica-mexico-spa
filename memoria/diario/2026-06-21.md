# Sesión Jun 21, 2026 — Bot bugs 2 + selección de día

## Lo que hicimos

### Bug 1: "Cancelar" → "No tienes citas próximas registradas"
- **Causa**: `.not("status","in","(cancelada,cancelado,no_show)")` — `cancelado` y `no_show` no son valores válidos del enum `appointment_status` → PostgreSQL lanza error de cast → Supabase devuelve `{ data: null }` → código asume 0 citas
- **Fix**: cambiar a `(cancelada,liberada)` — únicos valores válidos del enum para citas no-activas
- Aplica a 3 funciones: `verMiCita`, `iniciarCancelacionCita`, `iniciarReagendarCita`
- **Lección nueva**: siempre verificar que los valores en `.not(...,"in",...)` son válidos para el tipo enum de la columna. Un valor inválido no lanza error del lado del cliente — falla silenciosamente devolviendo `data: null`.

### Bug 2: Link GCal mostraba 2:30 PM en vez de 8:30 AM (paciente)
- **Causa**: `buildGCalLink` usaba `d.getHours()` — Deno corre en UTC → para slot de 8:30 CDMX, `getHours()` devuelve 14
- **Fix**: `d.toLocaleString("sv-SE", { timeZone: "America/Mexico_City" }).replace(/-/g,"").replace(" ","T").replace(/:/g,"")`
- GCal interpreta fechas sin sufijo como hora local del usuario → necesita CDMX local, no UTC

### Bug 3: Evento del doctor en GCal a hora incorrecta
- **Causa**: `inicio.toISOString()` genera `"2026-06-22T14:30:00.000Z"` (UTC con Z) → Google Calendar API **ignora el campo `timeZone`** cuando el `dateTime` tiene offset UTC explícito → evento creado a las 14:30 UTC
- **Fix**: helper `toMexicoLocalISO()` usando `toLocaleString("sv-SE", { timeZone: "America/Mexico_City" })` → devuelve `"2026-06-22T08:30:00"` sin Z → GCal usa `timeZone: "America/Mexico_City"` → evento a las 8:30 AM CDMX
- Aplica a `createCalendarEvent` y `updateCalendarEvent`

### GCal errores silenciosos → ya visibles

- Fix: throw cuando `!resp.ok` con body del error → outer catch persiste en `appointments.gcal_last_error`

## Sesión Jun 21 (continuación) — `supabase db push` + limpieza post-audit

### Problema: historial de migrations corrupto (Lovable vs CLI)

Lovable aplica migrations directamente a la DB sin pasar por el CLI. El CLI tenía:
- Entradas remotas sin archivo local (Lovable) → `repair --status reverted`
- Archivos locales sin registro remoto → `repair --status applied`
- Versión `20260602` con formato corto (8 dígitos en vez de 14) → CLI no la empareja con el archivo local, siempre aparece como "remoto Y local" simultáneamente

**Solución para `20260602`**: mover `supabase/migrations/20260602_create_clinics_and_memberships.sql` a `supabase/scripts/` — el CLI lo ignora, la DB ya tiene el schema.

**Regla aprendida**: Supabase CLI solo empareja versiones de 14 dígitos (YYYYMMDDHHMMSS). Versiones cortas (8 dígitos) están en un limbo permanente — mover a scripts/.

### Errores de SQL en migrations (syntax no soportada en PostgreSQL real)

| Syntax usada | Error | Fix correcto |
|---|---|---|
| `CREATE POLICY IF NOT EXISTS "name" ON t` | `syntax error at or near "NOT"` | `DROP POLICY IF EXISTS "name" ON t;` luego `CREATE POLICY "name" ON t` |
| `ALTER TABLE t ADD CONSTRAINT IF NOT EXISTS name` | `syntax error at or near "NOT"` | `DO $$ BEGIN IF NOT EXISTS (SELECT 1 FROM pg_constraint WHERE conname='name') THEN ALTER TABLE t ADD CONSTRAINT name ...; END IF; END $$;` |
| `CREATE INDEX CONCURRENTLY` vía `db push` o `db query` | `cannot run inside a transaction block` / `cannot be executed within a pipeline` | Quitar `CONCURRENTLY`, usar `CREATE INDEX IF NOT EXISTS` dentro de `BEGIN/COMMIT` |

**Regla aprendida**: `CREATE POLICY IF NOT EXISTS` NO EXISTE en PostgreSQL (ninguna versión). Solo existe el patrón DROP IF EXISTS + CREATE.

**Regla aprendida**: `ADD CONSTRAINT IF NOT EXISTS` NO EXISTE en PostgreSQL estándar. Usar DO block con `pg_constraint` check.

**Regla aprendida**: `CREATE INDEX CONCURRENTLY` no puede correr dentro de transacciones ni pipelines. Supabase CLI siempre usa pipeline/transacción. Solución práctica: quitar CONCURRENTLY para tablas pequeñas (lock es breve, <1s en tablas de clínica).
